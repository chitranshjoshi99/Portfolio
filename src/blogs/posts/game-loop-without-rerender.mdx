React re-renders a component every time its state changes. That's exactly what
you want for a form, and exactly what you *don't* want for a game loop that
fires sixty times a second. If the snake's position lived in `useState`, every
frame would queue a re-render, React would reconcile the whole tree, and your
buttery 60fps would turn to soup.

The fix is to keep the loop's mutable state out of React entirely.

## Refs hold the state, React holds the canvas

A `useRef` gives you a stable, mutable box that survives re-renders without
triggering them. The whole game (snake body, direction, food, score) lives in
one ref object. React renders the `<canvas>` exactly once; the loop mutates the
ref and paints pixels directly. Try it (arrow keys / WASD, or tap):

<div className="blog-demo">

</div>

## One loop, paused when off-screen

A single `requestAnimationFrame` loop reads `now`, steps the simulation on a
fixed tick, and draws. Crucially it only runs while the component is `active`,
so an off-screen channel burns zero CPU.

```ts
useEffect(() => {
  if (!active) return;            // paused when inactive
  let raf = 0, last = 0;
  const TICK = 150;               // ms per simulation step
  function loop(now: number) {
    raf = requestAnimationFrame(loop);
    if (now - last >= TICK) { step(stateRef.current); last = now; }
    draw(ctx, stateRef.current);
  }
  raf = requestAnimationFrame(loop);
  return () => cancelAnimationFrame(raf);
}, [active]);
```

## Why this matters

Zero re-renders per frame means React never becomes the bottleneck. The only
work per frame is your own `step` + `draw`. The same pattern powers Pong, the
dino runner, and every other canvas toy in the Labs. Reach for it any time you
have an animation loop that React shouldn't be sitting in the middle of.
